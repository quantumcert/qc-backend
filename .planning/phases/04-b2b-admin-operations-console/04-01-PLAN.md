<threat_model>

## Trust Boundaries

- Boundary A: dashboard/admin request -> backend platform admin route.
- Boundary B: tenant API key role -> human platform/tenant admin membership.
- Boundary C: migration input from dashboard DB -> canonical backend tenant/user models.
- Boundary D: schema-derived balances/queues -> future operational flows.

## Threats

- **HIGH: Tenant API ADMIN vira Platform Admin por engano.** Mitigation:
  middleware e testes separados para `requirePlatformAdmin`; `ApiKeyRole.ADMIN`
  permanece role tecnica tenant-scoped.
- **HIGH: Cross-tenant data exposure via trusted body tenantId.** Mitigation:
  admin scope vem do actor/membership e Platform Admin explicito; Tenant Admin
  usa tenant do contexto, nao do payload.
- **HIGH: Backfill duplica usuarios ou associa owner errado.** Mitigation:
  modelos de migration run/checkpoint/record com chaves idempotentes antes da
  execucao real.
- **MEDIUM: Audit logs guardam segredo/payload sensivel.** Mitigation:
  campos de audit sao metadata/payloadHash/error sanitizado; testes de request
  audit entram no Plano 03.
- **MEDIUM: Schema push quebra compatibilidade atual.** Mitigation:
  manter campos legados (`isActive`, `ownerRef`, `isActive` de Device) e adicionar
  novos campos de forma compatível.
</threat_model>
