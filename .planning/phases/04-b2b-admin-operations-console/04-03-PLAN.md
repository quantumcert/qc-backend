<threat_model>

## Trust Boundaries

- Boundary A: raw API key generation -> one-time UI display.
- Boundary B: API-key authenticated tenant traffic -> request audit store.
- Boundary C: platform admin action -> tenant credential lifecycle.

## Threats

- **HIGH: Reexibir ou persistir segredo bruto de API key.** Mitigation:
  usar padrão hash existente, raw secret somente no retorno da criação/rotação,
  testes provam que listagem não contém segredo/hash.
- **HIGH: Tenant gera primeira chave sem aprovação Quantum.** Mitigation:
  create initial key exige Platform Admin e tenant status válido.
- **HIGH: Tenant suspenso continua operando por API key ativa.** Mitigation:
  `validateApiKey` exige `tenant.status === ACTIVE` além de `isActive`; suspensão
  bloqueia uso sem revogação automática.
- **HIGH: API key ganha permissão por string livre ou selector sem política.** Mitigation:
  catálogo canônico de escopos, validação no dashboard/backend e enforcement no
  `DiamondProxy` por selector mapeado.
- **HIGH: Request audit vaza payload sensível.** Mitigation:
  middleware persiste somente metadados sanitizados e testes procuram ausência
  de body/raw key.
- **MEDIUM: Audit falha derruba API tenant.** Mitigation:
  audit best-effort com erro logado/sanitizado e sem bloquear resposta.
- **MEDIUM: Rotação cria janela insegura.** Mitigation:
  estado/lineage `rotatedFromApiKeyId`, reason e audit obrigatórios.
</threat_model>
