<threat_model>

## Trust Boundaries

- Boundary A: local operator provisioning script -> Stellar testnet/Friendbot/Soroban CLI.
- Boundary B: unauthenticated DLT network result -> backend persistence.
- Boundary C: tenant-owned event -> chain transaction billing/isolation record.
- Boundary D: tenant targetChain decision -> exactly one adapter call and one persisted chain result.

## Threats

- **HIGH: Secret leakage in provisioning output/files.** Mitigation: print copy/paste env only to stdout, never write secrets to tracked files, and add tests that no file write path stores `STELLAR_AUTHORITY_SECRET_KEY`.
- **HIGH: Cross-tenant transaction attribution.** Mitigation: require tenantId in Stellar anchor options and test `ChainTransaction.create.data.tenantId`.
- **MEDIUM: Algorand regression while changing adapter options.** Mitigation: queue regression test for `targetChain = ALGORAND`.
- **MEDIUM: Stellar shortcut blocks Solana later.** Mitigation: keep queue and adapter options chain-neutral; test routing through `tenant.targetChain` and `DLTAdapterFactory`.
- **MEDIUM: Fake contract id accepted during failed deploy.** Mitigation: provisioning script throws explicit error when deploy tooling is missing or deployment fails.
  </threat_model>
